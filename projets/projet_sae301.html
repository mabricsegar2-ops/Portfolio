Présentation

Projet fil rouge du parcours VCOD, réalisé en équipe de trois (avec Lucas Li-King RATIARINAIVONJANAHARY et Robin LECULLEE) autour du cas Darties, enseigne de produits électroménager, son et vidéo. L'objectif : construire une chaîne décisionnelle de bout en bout, de la collecte des données jusqu'à l'analyse.

Quatre maillons s'enchaînent : collecte automatisée (scraping + web services), base de production MySQL, entrepôt de données décisionnel (flocon dérivé en étoile) et processus ETL en Python qui alimente l'entrepôt depuis la production, jusqu'aux analyses OLAP et à la simulation économique.

1 · Collecte automatisée des données

Pour enrichir le référentiel produits et convertir les montants, deux sources externes sont collectées automatiquement, puis sauvegardées en CSV/JSON afin de se prémunir d'un changement ou d'une indisponibilité du site/service.

  • Scraping (Selenium) · collecte des produits Electrodépôt pour les trois familles (son, vidéo, électroménager · ex. réfrigérateurs, hi-fi), avec attentes conditionnelles et navigation pilotée, puis structuration en CSV via pandas.
  • Web service (requests) · récupération des cours monétaires journaliers face à l'euro (unité de compte de l'entrepôt), exportés en exchange_rates.json et en CSV.
  • Définition des familles, catégories et sous-catégories retenues pour rattacher chaque produit collecté à la nomenclature de l'entrepôt.

2 · Base de production & entrepôt décisionnel

La modélisation a été faite sous Oracle SQL Developer Data Modeler (fichier .dmd) puis implémentée et alimentée sous MySQL, sur deux schémas distincts.

  • Schéma de production (OLTP) · base relationnelle complète (magasins, produits, pays, devises, cours, ventes…) créée et alimentée par scripts SQL.
  • Entrepôt décisionnel (OLAP) · conçu en flocon puis dérivé en étoile : tables de faits faits_ventes et faits_cours, dimensions dim_magasin, dim_produit (sous-catégorie → catégorie → famille), dim_pays, dim_devise et dimension temps.
  • Vues matérialisées simulées pour accélérer les agrégats d'analyse.

3 · Processus ETL en Python (luigi)

Le cœur du projet : un ETL orienté objet qui transfère et transforme les données de la production (OLTP) vers l'entrepôt (OLAP). Il s'appuie sur luigi (orchestration des tâches et de leurs dépendances), mysql-connector / SQLAlchemy pour l'accès aux bases et pandas pour la transformation. Il peut s'exécuter en mode interactif, en batch ou en tâche planifiée Windows (module schedule), et cible le serveur de production MySQL projetud de l'UBS.

  • ETL 1 · dimensions devises/cours (dim_devise, faits_cours), agrégation mensuelle des cours puis insertion/mise à jour dans l'OLAP.
  • ETL 2 · dimensions principales (dim_magasin, dim_produit, dim_pays…) et table de faits faits_ventes, chargées depuis les CSV.
  • ETL 3 · chargement des objectifs commerciaux depuis les CSV annuels (2019-2023) : conversion du mois en lettres → id_temps, récupération des clés id_magasin / id_produit, puis upsert des colonnes ventes_objectif, ca_objectif et marge_objectif.
# ETL 3 · conversion du mois (en lettres) vers une clé de dimension temps
MOIS_MAP = {
    "JANVIER": "01", "FÉVRIER": "02", "MARS": "03", "AVRIL": "04",
    "MAI": "05", "JUIN": "06", "JUILLET": "07", "AOÛT": "08",
    "SEPTEMBRE": "09", "OCTOBRE": "10", "NOVEMBRE": "11", "DÉCEMBRE": "12"
}

def get_id_temps(annee, mois_lettres):
    mois = MOIS_MAP[mois_lettres.strip().upper()]
    return str(annee) + mois   # ex. (2023, "MARS") -> "202303"

4 · Simulation économique & analyses OLAP

Côté restitution, des scripts SQL et procédures stockées MySQL assurent la simulation économique et les requêtes d'analyse OLAP sur l'entrepôt (croisements ventes/objectifs, agrégats par famille de produits, conversions de devises via faits_cours) · la base prête pour un cube et un reporting décisionnel.

import luigi, mysql.connector
from datetime import datetime
from config import MYSQL_OLTP_CREDENTIALS, MYSQL_OLAP_CREDENTIALS

class ExtractFromOLTP(luigi.Task):
    def output(self):
        return luigi.LocalTarget('data.txt')

    def run(self):
        cnx = mysql.connector.connect(**MYSQL_OLTP_CREDENTIALS)
        cursor = cnx.cursor()
        with open('last_update_luigi.txt') as f:
            last = datetime.strptime(f.read().strip(), '%Y-%m-%d %H:%M:%S')
        # Extraction incrementale : seulement les lignes modifiees
        cursor.execute("""
            SELECT DATECOM, ARTICLE, CLIENT,
                   SUM(QUANTITE), SUM(PRIX*QUANTITE)
            FROM Commandes
            WHERE update_date > %s
            GROUP BY DATECOM, ARTICLE, CLIENT
        """, (last,))

Apprentissages critiques acquis

Scraping, web services, CSV annuels et bases MySQL : l'ETL nettoie, harmonise et intègre ces sources variées dans un modèle décisionnel cohérent.

L'ETL en POO orchestré par luigi, exécutable en interactif, en batch ou en tâche planifiée, m'a fait concevoir une chaîne de traitement automatisée, reproductible et maintenable.

Oracle SQL Developer Data Modeler pour la modélisation, MySQL pour les schémas, Selenium/requests pour la collecte, luigi/SQLAlchemy pour l'ETL : chaque outil mobilisé à sa juste place.
